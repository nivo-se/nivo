+# Nivo Senior Codebase Audit (2026-03-24)
+
+## Scope
+- Full-repo architecture and code-path review with special focus on screening/profile/ranking/filtering workflows.
+- Traced frontend -> API -> service -> DB for Universe, screening profiles, campaigns, shortlist and list creation.
+- Reviewed deep-research boundary documents and selected backend modules for architecture drift signals.
+
+## 1) Executive summary
+
+### Critical issues
+1. Screening profile configs are accepted as arbitrary JSON with no schema validation, so malformed configs silently degrade to `None` score/ranking behavior.
+2. The DB schema explicitly allows multiple active profile versions per profile; APIs then do `LIMIT 1` without deterministic tie-break, making active config selection nondeterministic.
+3. Layer 0 ranking uses only `profile_weighted_score`; rows with missing score become `-inf` and are effectively random tail members with no explicit exclusion reason.
+4. Layer 3 assigns final ranks to ineligible rows (`out_of_mandate`, rejected uncertain) with `combined_score=0` instead of null rank, confusing “ranked” vs “eligible”.
+5. Campaign candidates list API sorts by `layer0_rank`, not `final_rank`, so UI can display stale ordering after Layer 3.
+6. Multiple API paths silently return empty arrays on backend errors (`/api/universe/query`, `/api/lists`), masking production failures as “no results”.
+7. Legacy `/api/filters` endpoint computes scores via pandas on in-memory snapshots (LIMIT 5000), producing nondeterministic shortlist behavior unrelated to campaign flow.
+8. Create/update endpoints often “read latest row by owner + created_at desc” instead of using `RETURNING`, opening race-window record mixups under concurrent requests.
+
+### High-value improvements
+9. Introduce typed screening profile schema (pydantic/jsonschema), with strict validation and canonical normalization before write.
+10. Move all Layer 1 deterministic ranking/filtering into SQL-native, testable functions with explicit handling for missing metrics.
+11. Add “rank explainability payload” (per-variable contribution + missing-data penalties + exclusion flags) and surface this in UI.
+12. Collapse duplicate screening surfaces (`screening campaigns` vs `screening_runs shortlist`) into one clear product path.
+13. Replace row-by-row DB updates in campaign stages with bulk UPSERT/CTE updates to reduce write amplification and lock contention.
+14. Add contract tests for frontend/backend payload compatibility across universe query, profile config, candidates view, and list-from-query.
+
+### Screening-specific findings
+15. Determinism is partly good (Layer 0 SQL + Layer 3 blend), but active-profile ambiguity and silent fallbacks break reproducibility.
+16. Weighted scoring currently skips missing variables instead of penalizing uncertainty, which can over-promote sparse companies.
+17. Layer 2 `layer2Limit` exists in API schema but is not enforced in execution.
+18. Save-list-from-query does not apply profile-driven exclusion/weights when profile context is active in UI, causing “what user saw” != “what was saved”.
+19. Confidence bucketing in persisted `screening_runs` shortlist is inferred heuristically from JSON payload shape, not canonical run-time fields.
+20. The screening campaign workflow lacks idempotency guards/locks to prevent overlapping stage starts per campaign.
+
+## 2) System map (implemented, not idealized)
+- **Frontend app**: React/Vite UI under `frontend/src`, using typed service clients in `frontend/src/lib/api/*` and additional legacy service layers.
+- **Backend API**: FastAPI monolith (`backend/api/main.py`) registering many routers for universe, screening, campaigns, lists, enrichment, deep research.
+- **DB layer**: single `PostgresDBService` with raw SQL helper; it converts `?` placeholders to `%s` and commits per query call.
+- **Jobs**: Redis + RQ worker in docker-compose for enrichment/deep_research/ai_analysis queues.
+- **Auth**: JWT middleware (Auth0 RS256 via JWKS) when `REQUIRE_AUTH=true`; otherwise many endpoints fallback to hardcoded dev user.
+- **Screening deterministic path (Layer 1)**:
+  - UI -> `/api/universe/query` for filter/sort/pagination.
+  - Optional profile config loaded from `screening_profile_versions` active version.
+  - Score computed in Python row materialization using config `variables`+`weights`.
+- **Screening orchestrator path**:
+  - `/api/screening/campaigns` create/start/layer1/layer2/layer3.
+  - Layer0 calls `execute_universe_query(max_rows=None)` then sorts in Python by `profile_weighted_score`.
+  - Layer1/2 are LLM-gated steps; Layer3 blends deterministic + fit scores.
+- **Saved lists path**:
+  - `/api/lists/from_query` rebuilds universe SQL and inserts list items.
+- **Deep research path**:
+  - Separate agentic orchestration modules plus persistence models.
+  - Some routes/docs still indicate stubs/placeholders.
+
+### Boundary quality
+- **Clean boundaries**: explicit screening campaign stage tables; profile versioning tables; separate deep-research persistence modules.
+- **Leaky boundaries**:
+  - Legacy and new screening flows coexist (`/api/filters`, campaigns, screening_runs shortlist).
+  - Layer 1 deterministic logic split across SQL + Python materialization and reused inconsistently.
+  - UI service layers duplicate adapters and partially ignore backend fields (e.g., `profileId` in some flow paths).

+## 3) Full codebase audit
+
+### Architecture
+1. **[critical][architecture/screening]** Multiple active profile versions allowed by schema comments and index setup.
+   - Why: Active profile resolution uses `LIMIT 1` and can select arbitrary row.
+   - Fix: unique partial index `UNIQUE(profile_id) WHERE is_active`; transactional activate endpoint.
+   - Size: medium refactor.
+
+2. **[high][architecture]** Competing screening systems (`campaigns`, `screening_runs`, legacy `filters`) create contradictory outputs.
+   - Why: users can see different “top companies” depending on entrypoint.
+   - Fix: deprecate `/api/filters`; designate one production path and gate others behind feature flags.
+   - Size: larger redesign (product + API migration).
+
+3. **[medium][architecture]** Deep-research modules still include documented stubs; architecture docs and implementation drift.
+   - Why: operational confusion; false assumptions during integration.
+   - Fix: mark all stub routes in OpenAPI with explicit `x-status: experimental/stub`; tighten docs.
+   - Size: quick win.
+
+### Backend/API
+4. **[critical][screening/api]** No server-side schema validation for profile `config_json` on create version.
+   - Why: invalid variables/weights silently accepted; ranking breaks quietly.
+   - Fix: pydantic model for profile config + strict enum/field whitelist + normalization.
+   - Size: medium refactor.
+
+5. **[high][api]** Silent fallback to empty result on query errors in universe endpoint.
+   - Why: hides production regressions as “0 matches”.
+   - Fix: return 5xx with structured error and correlation id; log full stack once.
+   - Size: quick win.
+
+6. **[high][api]** Create endpoints fetch “latest row by owner” after insert instead of using `RETURNING`.
+   - Why: race conditions under concurrent writes.
+   - Fix: use `INSERT ... RETURNING` and transaction-scoped response.
+   - Size: quick win.
+
+7. **[medium][api]** `list_candidates` orders by `layer0_rank` always.
+   - Why: post-layer3 UI does not show true final ordering.
+   - Fix: order by `final_rank NULLS LAST, layer0_rank`; add query parameter for sort mode.
+   - Size: quick win.
+
+8. **[medium][api]** `layer2Limit` exists in schema but not applied in layer2 execution.
+   - Why: unbounded token/cost compared to expected campaign settings.
+   - Fix: apply `LIMIT` in `_candidates_for_layer2` using resolved params.
+   - Size: quick win.
+
+9. **[medium][api/screening]** Campaign stage start endpoints allow concurrent starts with no advisory lock.
+   - Why: duplicate writes and inconsistent stage status under repeated clicks.
+   - Fix: `pg_advisory_xact_lock(hash(campaign_id,stage))` + state precondition checks.
+   - Size: medium refactor.
+
+10. **[medium][api/lists]** `/from_query` rebuilds query without profile context scoring/exclusions from active UI profile unless manually included.
+    - Why: list contents can diverge from what user reviewed in universe profile mode.
+    - Fix: include `profileId/profileVersionId` in payload and reuse `execute_universe_query` path.
+    - Size: medium refactor.
+
+### Frontend/UI
+11. **[high][frontend/screening]** Universe service adapter sets `latest_year` to current year, ignoring backend field.
+    - Why: UI may display wrong fiscal recency confidence.
+    - Fix: pass backend `latest_year` through mapping.
+    - Size: quick win.
+
+12. **[medium][frontend/api]** Type definitions for list scope omit `public` while backend supports it.
+    - Why: contract drift and hidden feature path bugs.
+    - Fix: align union types and forms with backend enum.
+    - Size: quick win.
+
+13. **[medium][frontend]** Multiple service layers (`lib/services/*` and `lib/api/*`) duplicate transformations.
+    - Why: maintenance overhead and divergence risk.
+    - Fix: consolidate to one typed API client layer.
+    - Size: medium refactor.
+
+### Database/schema/data integrity
+14. **[critical][db/screening]** Active version uniqueness not enforced at DB level.
+    - Fix: migration adding partial unique index + cleanup script.
+    - Size: medium refactor.
+
+15. **[high][db]** Campaign candidate schema lacks explicit check constraints for score ranges.
+    - Why: invalid values can be persisted (e.g., negative fit or >100).
+    - Fix: add CHECK constraints and sanitize writes.
+    - Size: medium refactor.
+
+16. **[medium][db]** Heavy use of JSONB payloads for operationally-critical fields (evidence, reasons) without extracted indexed columns.
+    - Why: poor queryability for analytics and QA.
+    - Fix: denormalize key fields (confidence, reason code arrays) into typed columns.
+    - Size: larger redesign.
+
+### Async/job/orchestration flows
+17. **[high][jobs/screening]** Layer 0 and Layer 3 perform per-row update loops.
+    - Why: O(n) round-trip DB writes; slow on larger campaign sizes.
+    - Fix: bulk update via temp table + join/CTE.
+    - Size: medium refactor.
+
+18. **[medium][jobs]** Campaign status transitions are non-atomic across multiple updates.
+    - Why: partial stage failures can leave stale status/state.
+    - Fix: explicit transactions per stage; state machine helper.
+    - Size: medium refactor.
+
+### Auth/security
+19. **[medium][auth]** Dev-user fallback is used in many endpoints whenever auth disabled.
+    - Why: accidental shared writes in non-prod; hard-to-trace ownership.
+    - Fix: configurable dev user seed per environment + audit logs + warning headers.
+    - Size: quick win.
+
+20. **[low][auth]** Public-path checks and CORS checks duplicate allowlists in separate files.
+    - Why: policy drift risk.
+    - Fix: centralize CORS/auth config.
+    - Size: medium refactor.
+
+### Testing/observability
+21. **[high][tests]** Screening orchestrator lacks integration tests for end-to-end deterministic behavior and ranking reproducibility.
+22. **[high][tests]** No contract tests ensuring frontend candidate ordering/fields match backend semantics post-layer3.
+23. **[medium][observability]** Stage stats do not include missing-data rate, invalid-score count, or exclusion reasons distribution.
+
+### Performance
+24. **[high][performance]** Universe source subquery computes many derived fields inline for each request; no materialized helper for hot filtering dimensions.
+25. **[high][performance]** `/api/filters` pandas pipeline loads up to 5000 rows and computes in Python for each call.
+26. **[medium][performance]** `table_exists` and relation-column checks in request path can add avoidable metadata queries.
+
+### Developer experience / maintainability
+27. **[high][DX]** Mixed naming and historical compatibility layers make it hard to identify canonical screening path.
+28. **[medium][DX]** Docs reference architecture boundaries that code only partially enforces.
+29. **[medium][DX]** Lack of explicit “deprecated endpoint” markers for legacy screening routes.

+## 4) Dedicated screening service review
+
+### 4.1 Direct answers to required questions
+
+1. **Is current screening architecture logically sound?**
+   - **Partially.** Core idea (deterministic Layer 0 + optional AI gating later) is right, but implementation has reproducibility gaps (active-version ambiguity, silent fallbacks, missing-data handling).
+
+2. **Where is screening too brittle?**
+   - Profile config acceptance (no validation), active-version selection, API silent failures, and ranking behavior when profile score missing.
+
+3. **Where can scoring become misleading/noisy?**
+   - Missing variables are skipped (not penalized), default score normalization caps values to [0,1] without data-quality weighting, Layer 3 normalizes only by cohort max (susceptible to outliers).
+
+4. **Hidden mismatches between profile config/backend/frontend display?**
+   - Yes: frontend list/save and service adapters don’t consistently carry profile context and some mapped fields are overwritten/defaulted (e.g. latest year).
+
+5. **Deterministic rules that should replace heuristics?**
+   - Confidence buckets, missing-data treatment, and shortlist eligibility should be explicit rules/columns, not inferred from JSON blobs.
+
+6. **Data-model or SQL issues distorting ranking?**
+   - Multiple active profile versions, candidates ordered by layer0 rank in listing, and final_rank assigned to ineligible rows.
+
+7. **Can implementation produce trustworthy top-ranked lists now?**
+   - **For controlled happy-path data: yes-ish.** For production-grade trust: **not yet** due to above determinism and explainability gaps.
+
+8. **Top 10 changes to improve screening quality**
+   1) Validate profile config schema on write.
+   2) Enforce single active version in DB.
+   3) Add explicit missing-data penalties in score computation.
+   4) Persist per-variable score contributions for explainability.
+   5) Apply `layer2Limit` and consistent eligibility rules.
+   6) Null final_rank for ineligible rows.
+   7) Sort candidates by final_rank post-layer3.
+   8) Replace silent empty responses with explicit errors.
+   9) Ensure save-from-query uses same profile context as viewed universe.
+   10) Add deterministic regression snapshot tests for top-N outputs.
+
+9. **Top 10 changes to improve performance + maintainability**
+   1) Bulk UPSERT updates in layer stages.
+   2) Add materialized/helper view for hot universe fields.
+   3) Remove `/api/filters` pandas shortlist path from prod.
+   4) Consolidate frontend API adapters.
+   5) Cache metadata checks (table/column exists) at startup.
+   6) Add stage-level advisory locks.
+   7) Precompute profile score in SQL for large campaigns.
+   8) Add campaign-stage state machine utility.
+   9) Add indexed typed columns for key JSON metrics.
+   10) Reduce duplicated query-building code paths.
+
+10. **What should be postponed?**
+   - Online learning loops that alter Layer 1 scoring automatically.
+   - Rich diversification optimizer in shortlist selection.
+   - Complex probabilistic ranking before deterministic baseline + explainability are stable.
+
+### 4.2 Additional hard calls
+- Keep LLM out of Layer 1 ranking logic; only use LLM for downstream relevance/fit after deterministic candidate reduction.
+- Add a strict “screening run manifest” artifact per campaign stage (input hash, profile version, SQL fingerprint, result checksum).
+- Stop exposing legacy shortlist endpoints as equal peers in UI navigation.
